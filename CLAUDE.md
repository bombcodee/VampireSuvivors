# CLAUDE.md - 프로젝트 지시서

> 모든 코드 생성, 수정, 리뷰 시 이 문서를 참고합니다.
>
> **⚠️ 작업 시작 전 반드시 함께 확인할 문서:**
> - `CONVENTIONS.md` — 코드 작성 스타일 규칙 (네이밍, 들여쓰기, 금지사항 등)
> - `ARCHITECTURE.md` — 시스템 구조, 상태 머신, 데이터 흐름, 파일 의존성

## 프로젝트 개요

- **프로젝트명**: Vampire Survivors 클론 → 추후 오리지널 IP로 발전 예정
- **목적**: 게임 개발의 전체 사이클 체험 (기획→개발→배포→운영→유지보수→수익화)
- **최종 목표**: 실제 Steam/모바일 출시 및 수익화까지 경험
- **기술 스택**: 순수 HTML5 Canvas + JavaScript (ES6+)
- **타겟 플랫폼**: 웹 브라우저 (1차) → Steam/모바일 (확장)
- **개발자 수준**: 게임 개발 입문자 - 코드에 한글 주석을 상세히 작성할 것
- **상세 비전/로드맵**: `docs/00_프로젝트_비전_및_로드맵.md` 참고

## 기술 규칙

### 언어 & 환경
- JavaScript ES6+ (모듈 시스템 사용: import/export)
- HTML5 Canvas API로 렌더링
- 외부 라이브러리/프레임워크 사용 금지 (순수 JS로만)
- 번들러 없음 (Webpack, Vite 등 사용하지 않음)
- `<script type="module">` 로 모듈 로딩

### 코드 스타일
- 파일명: PascalCase (예: `Player.js`, `EnemySpawner.js`)
- 클래스명: PascalCase (예: `class Player`, `class WeaponSystem`)
- 변수/함수명: camelCase (예: `moveSpeed`, `takeDamage()`)
- 상수: UPPER_SNAKE_CASE (예: `MAX_ENEMIES`, `PLAYER_SPEED`)
- 게임 설정값은 `src/data/` 폴더의 JSON 또는 config 파일에서 관리
- 매직 넘버 금지 → 상수로 정의

### 주석 규칙
- 모든 클래스 상단에 한글 설명 주석 작성
- 모든 메서드 상단에 역할 설명 주석 작성
- 복잡한 로직에는 인라인 주석 추가
- 예시:
```javascript
/**
 * 플레이어 클래스
 * - 이동, 체력, 피격, 사망을 관리한다
 * - WASD 또는 방향키로 조작
 */
class Player {
    /**
     * 플레이어를 이동시킨다
     * @param {number} dt - 델타타임 (프레임 간 시간 간격)
     */
    move(dt) {
        // 대각선 이동 시 속도 정규화
        ...
    }
}
```

## 아키텍처 규칙

### 폴더 구조
```
src/
├── core/           # 게임 엔진 핵심 (Game, Loop, Input, Camera)
├── entities/       # 게임 객체 (Player, Enemy, Projectile, Gem)
├── weapons/        # 무기 시스템 (Weapon, 각 무기별 클래스)
├── systems/        # 게임 시스템 (Spawn, Exp, Upgrade, Stage)
├── ui/             # UI 컴포넌트 (HUD, LevelUpUI, MenuUI)
├── data/           # 데이터 정의 (무기 스탯, 적 스탯, 설정값)
└── utils/          # 유틸리티 (수학 함수, 충돌 판정, 오브젝트풀)
```

### 모듈화 원칙 (조립식 설계) ⚠️ 모든 코드에 적용
- 모든 기능은 독립된 모듈(클래스/파일)로 구현한다 — 추가·수정·제거가 다른 코드에 영향 최소화
- 모듈 간 통신은 Game 인스턴스 또는 명확한 인터페이스를 통해서만 한다
- 기존 코드 수정 시에도 모듈 경계를 유지한다 — 한 모듈 안에서 해결할 수 있으면 밖을 건드리지 않는다
- config.js에 데이터를 모으고, 로직은 각 모듈이 독립적으로 처리한다

### 설계 패턴
- **Game 클래스**: 싱글톤. 게임 전체 상태를 관리하는 중앙 매니저
- **Entity 패턴**: 모든 게임 객체는 `x, y, width, height, active` 속성을 가짐
- **시스템 분리**: 로직(System)과 표현(Render)을 분리
- **오브젝트 풀링**: 적, 투사체, 보석 등 대량 생성 객체는 풀링 사용
- **데이터 주도 설계**: 무기/적/아이템 스탯은 코드가 아닌 데이터 파일로 관리

### 파일 작성 원칙
- 1 파일 = 1 클래스 (파일명 = 클래스명)
- 한 파일은 하나의 역할에 집중한다 (코어 파일은 길어질 수 있음)
- 분리 시점: "이 코드가 이 파일의 역할인가?" → NO면 분리
- 순환 참조(circular dependency) 금지
- 전역 변수 사용 금지 (Game 인스턴스를 통해 접근)

## 개발 프로세스

### 작업 순서 (⚠️ 순서 엄수)
```
1. [준비] CONVENTIONS.md, ARCHITECTURE.md, 로드맵(00_프로젝트_비전_및_로드맵.md) 확인
      → 방향이 바뀌면 로드맵을 먼저 업데이트한 후 진행
2. [설계] 새 기능이면 → 시스템 구조 설명 후 사용자 확인받기
3. [개발] 코드 작성 (한글 주석 포함)
4. [검증] 테스트 가능한 상태로 만들기 (브라우저에서 즉시 확인)
5. [수정] 버그 발견 시 → 원인 분석 먼저, 수정 후 설명
6. [문서] 개발 완료 후 맨 마지막에 → 문서 업데이트 여부를 사용자에게 질문
         "문서 업데이트할까요?" 라고 물어보고, 허가 시에만 진행
```
> **핵심 원칙**: 개발 흐름을 끊지 않는다. 문서 정리는 항상 맨 마지막.
> 코드 작성 중간에 문서를 건드리지 않는다.

### 문서 관리 규칙 (⚠️ 사용자 확인 필수)
> 아래 항목은 자동 승인(auto-approve)과 무관하게 **반드시 사용자 허가를 받고** 실행한다.

**사용자 확인이 필요한 작업:**
- docs/ 폴더 내 문서 신규 생성
- docs/ 폴더 내 기존 문서의 구조 변경 (섹션 추가/삭제/이동)
- CLAUDE.md, ARCHITECTURE.md, CONVENTIONS.md 수정
- CHANGELOG.md에 새 버전 항목 추가
- 버그 트래커(06_버그_트래커.md) 상태 변경 (OPEN → FIXED 등)
- 개발 예정 사항(07_개발_예정_사항.md) 상태 변경 (PLANNED → DONE 등)

**사용자 확인 없이 가능한 작업:**
- 코드 내 한글 주석 추가/수정
- config.js 수치 조정 (밸런스 튜닝)
- 기존 패턴을 따르는 새 코드 파일 생성 (예: 새 무기 클래스)
- 버그 수정 (코드 변경)

**문서 업데이트 시점:**
- 새 기능 완성 후 → CHANGELOG.md, 07_개발_예정_사항.md 업데이트 제안
- 버그 발견/수정 후 → 06_버그_트래커.md 업데이트 제안
- Phase 전환 시 → 00_비전_로드맵.md 체크리스트 업데이트 제안

### 커밋 메시지 규칙
```
feat: 새 기능 추가 (예: feat: 플레이어 이동 구현)
fix: 버그 수정 (예: fix: 대각선 이동 속도 정규화)
refactor: 코드 구조 개선 (예: refactor: 무기 시스템 클래스 분리)
docs: 문서 수정
asset: 리소스 추가/수정
```

## 게임 특화 규칙

### 좌표계
- Canvas 좌표: 좌상단이 (0, 0), 오른쪽이 +x, 아래가 +y
- 게임 월드 좌표: 카메라 오프셋 적용하여 변환

### 성능 가이드
- 화면에 적 200마리 이상일 때도 60fps 유지가 목표
- `requestAnimationFrame` 사용 (setInterval 금지)
- Canvas 드로우콜 최소화 (배치 렌더링)
- 화면 밖 객체는 업데이트/렌더 스킵

### 밸런스 데이터
- 모든 수치(체력, 공격력, 속도 등)는 `src/data/` 에서 관리
- 밸런스 조정 시 코드 수정 없이 데이터만 변경

## 소통 규칙

- 입문자에게 설명하듯이 한글로 소통
- 새로운 프로그래밍 개념 사용 시 간단히 설명 추가
- 한 번에 너무 많은 코드를 생성하지 말고, 단계별로 진행
- 각 단계가 실행 가능한 상태를 유지 (깨진 빌드 금지)
- 작업 완료 후 사용자의 원래 프롬프트를 요약 정리하여 마지막에 보여준다
  - 형식: 완료된 내용 나열 후, "**프롬프트 요약:**" 항목으로 사용자가 요청한 내용을 번호 리스트로 정리
  - 목적: 사용자가 자신의 요청이 빠짐없이 처리되었는지 검토할 수 있도록

## 토큰 효율성 규칙

- CLAUDE.md와 CONVENTIONS.md 간 중복 내용이 많아지면 사용자에게 알리고 정리 방안 제안
- 문서 간 불필요한 중복이 발견되면 "이 섹션을 CONVENTIONS.md로 이관할까요?" 등 대안 제시

## MEMORY.md 자동 저장 규칙

> MEMORY.md = 현재 진행 상태 포스트잇 (200줄 이내, 최신 상태로 덮어쓰기)
> 위치: .claude/projects/.../memory/MEMORY.md (프로젝트별 자동 분리)

**자동 저장 시점 (세이브 포인트):**
- 버그 수정 완료 후
- 새 기능 구현 완료 후
- 중요한 설계 결정 확정 후
- Phase 전환 시
- 대화 마무리 시
- 사용자가 "저장해줘" 요청 시

**저장 알림:**
- 저장할 때마다 사용자에게 알림
- 단, 코딩/작업 중에는 알리지 않고, 작업 완료 후 [문서] 단계에서 모아서 알림

**저장 내용 (간결하게):**
- 현재 Phase 및 진행 상태
- 다음에 할 작업
- 사용자가 내린 주요 결정
- 발견된 이슈나 보류 사항
